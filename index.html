The Big Picture What's changing, why it matters, and the scope of impact
Overview

Salesforce's "Secure by Default" Shift

For years, Apex database operations ran in system mode by default — silently bypassing object and field-level security. Summer '26 flips that default and removes the legacy clause that partially addressed it.

🔐

User Mode by Default

All Apex DB operations — SOQL, SOSL, DML, Database methods — now respect the running user's FLS, OLS, and sharing rules out of the box at API 67.0.

🚫

WITH SECURITY_ENFORCED Removed

Classes on API 67.0+ that still use WITH SECURITY_ENFORCED will fail to compile. Not a warning — a hard error.

✅

One Fix for Both

Replace WITH SECURITY_ENFORCED with WITH USER_MODE. That one swap resolves the compile error and aligns with the new default behaviour.

02
Database Operations Run in User Mode by Default Understanding the mode shift and how to adapt
Change 1 of 2

The Default Mode Flip

Any Apex class compiled at API 67.0+ will default all database operations to user mode. This enforces field-level security, object permissions, and sharing rules — no additional code required.

⚠️
Preview Notice: This is a versioned change. Classes remaining on API ≤ 66.0 are completely unaffected. Only bump API versions intentionally for classes that you have tested and updated.
AccountQuery.cls
−// ❌ API ≤ 66.0: implicit system mode — bypasses FLS and OLS Account acc = [SELECT Id, Name, AnnualRevenue__c FROM Account WHERE Name = 'Acme' LIMIT 1];
+// ✅ API 67.0+: explicit user mode (enforces permissions) Account acc = [SELECT Id, Name FROM Account WHERE Name = 'Acme' WITH USER_MODE LIMIT 1];
+// ✅ When system-level access is truly needed (must be intentional) Account acc = [SELECT Id, Name FROM Account WHERE Name = 'Acme' WITH SYSTEM_MODE LIMIT 1];
💡
At API 67.0, any SOQL written without an explicit clause behaves the same as writing WITH USER_MODE. The recommendation is to always be explicit — it signals intent during code review and is immune to future default changes.
AccountService.cls
−update a; // ❌ implicit system mode
−insert newAccount; // ❌ implicit system mode
+update as user a; // ✅ explicit user mode
+insert as user newAccount; // ✅ explicit user mode
+delete as system oldLog; // ✅ explicit system — only when justified
Full DML Example 
Account a = [SELECT Id, Name
   FROM Account
   WHERE Support__c = 'Premier'
   WITH USER_MODE];

a.Rating = 'Hot';
update as user a;
DynamicQuery.cls
−// ❌ No AccessLevel — implicit system mode Database.query('SELECT Id, Name FROM Account WHERE Rating = \'Hot\'');
+// ✅ Explicit user mode List<Account> results = Database.query( 'SELECT Id, Name FROM Account WHERE Rating = \'Hot\'', AccessLevel.USER_MODE );
+// ✅ Search methods — same pattern List<List<SObject>> sr = Search.query( 'FIND {Acme} IN ALL FIELDS RETURNING Account(Id, Name)', AccessLevel.USER_MODE );

Key Behaviours & Gotchas

If you write WITH USER_MODE on a query inside a class declared without sharing, the operation still respects the running user's sharing rules. User mode is a stronger signal than the class-level without sharing keyword. This can expose fewer records than expected to privileged service classes.
With WITH SYSTEM_MODE, field-level and object-level access is bypassed, but record-level visibility is controlled by the enclosing class's with sharing / without sharing / inherited sharing declaration — not the user.
This is a versioned change. Apex classes compiled at API version 66.0 or below continue to default to system mode, exactly as before. You control the upgrade — migrate and test each class intentionally.
Salesforce recommends always writing WITH USER_MODE or WITH SYSTEM_MODE explicitly — even in classes already on API 67.0. Explicit intent makes security reviews trivial and protects against future default changes.
03
WITH SECURITY_ENFORCED is Removed Hard compile error at API 67.0 — and why the replacement is better
Change 2 of 2

Compile Error — Not a Warning

Apex classes set to API version 67.0 or later that include WITH SECURITY_ENFORCED will refuse to compile. Your deployment will fail. This is by design — the replacement is WITH USER_MODE.

🚨
This is your highest-priority fix. Runtime errors can be caught in testing. A compile error blocks deployment entirely. Find and replace every instance before upgrading API versions.
The fix — one keyword swap
−// ❌ API 67.0+: COMPILE ERROR Account acc = [SELECT Id FROM Account WHERE Name = 'Acme' WITH SECURITY_ENFORCED LIMIT 1];
+// ✅ API 67.0+: compiles and enforces full user permissions Account acc = [SELECT Id FROM Account WHERE Name = 'Acme' WITH USER_MODE LIMIT 1];

Bonus: See All FLS Errors at Once

With WITH SECURITY_ENFORCED you only ever saw the first inaccessible field. With WITH USER_MODE you can inspect every inaccessible field in one go using getInaccessibleFields().

FLS error inspection with USER_MODE 
try {
    List<Account> accs = [
        SELECT Id, Name, AnnualRevenue, Weird_Field__c
        FROM Account
        WITH USER_MODE
    ];
} catch (QueryException e) {
    // Returns ALL fields the user cannot access, not just the first
    Map<String, Set<String>> blocked = e.getInaccessibleFields();
    for (String obj : blocked.keySet()) {
        System.debug('Object: ' + obj + ' | Blocked fields: ' + blocked.get(obj));
    }
}

5 questions covering both changes. Answers are explained as you go.

Question 1 of 5At API version 67.0, what happens if you deploy an Apex class that contains the clause WITH SECURITY_ENFORCED?
A
It runs but logs a deprecation warning
B
The class fails to compile — deployment is blocked
C
It runs in user mode automatically
D
It only fails at runtime, not compile time
Correct — B. WITH SECURITY_ENFORCED is removed from the Apex compiler in API 67.0. Any class referencing it will fail to compile entirely. This is not a runtime warning — it blocks deployment.
Question 2 of 5A developer has a SOQL query in a class declared without sharing. They add WITH USER_MODE to the query. What does this change?
A
Nothing — without sharing takes precedence
B
Only field-level security is enforced; record-level is still bypassed
C
The query now enforces the running user's sharing rules, overriding the class declaration
D
It causes a compile error inside a without sharing class
Correct — C. WITH USER_MODE on an operation is a stronger signal than the class-level without sharing declaration. The query will respect the running user's record-level sharing rules regardless of the class's sharing setting.
Question 3 of 5Which advantage does WITH USER_MODE have over WITH SECURITY_ENFORCED regarding FLS errors?
A
USER_MODE skips FLS checks entirely for better performance
B
USER_MODE surfaces all FLS errors; SECURITY_ENFORCED only surfaces the first
C
USER_MODE throws a different exception type
D
They behave identically for FLS
Correct — B. WITH SECURITY_ENFORCED halts on the very first inaccessible field. WITH USER_MODE continues evaluating and collects all inaccessible fields, which you can then inspect via QueryException.getInaccessibleFields().
Question 4 of 5Which of the following correctly sets user mode on a Database.query() call?
A
Database.query(soql, 'USER_MODE')
B
Database.query(soql WITH USER_MODE)
C
Database.query(soql, AccessLevel.USER_MODE)
D
Database.queryAsUser(soql)
Correct — C. Database and Search methods accept an AccessLevel enum parameter as their second argument. The valid values are AccessLevel.USER_MODE and AccessLevel.SYSTEM_MODE.
Question 5 of 5A team has 50 Apex classes all on API version 64.0. They upgrade two of them to API 67.0. What is the impact on the other 48 classes?
A
They all automatically inherit the new user-mode default
B
They fail at runtime on the next execution
C
They are completely unaffected — the change is versioned per class
D
They generate compiler warnings but continue to work
Correct — C. This is a versioned change. Each Apex class independently governs its own API version. Classes on API 64.0 continue to default to system mode as before. Only classes explicitly bumped to API 67.0+ are subject to the new rules.
